# KhaoJai — Feature Specification v1.0

> 專為中文母語者設計的泰語閱讀工具。用戶貼入泰文，App 即時分解每個詞的聲音與聲調，以視覺化方式呈現。無課程、無打卡、純工具。

---

## 產品定位

**目標用戶**：中文（普通話或粵語）母語者，希望閱讀或理解泰文的人。
**核心假設**：中文母語者學習泰語具有天然優勢——語序相近、音調系統類似——但現有工具都按英語母語者設計，完全浪費這個優勢。
**設計原則**：永遠說聲音，不說分類。「像『可』的 kh-」比「高類輔音」有用一百倍。

---

## Feature 1：文字輸入與解析

### 概述
這是整個 App 的入口。用戶將任意泰文文字貼入輸入框，App 呼叫 AI (OPENAI) API 同時完成分詞與語言分析，回傳結構化 JSON，再由前端渲染成可互動的閱讀介面。

### 輸入方式
- **主要方式：直接貼入泰文文字**
  - 支援任意長度（短句、長段落、單詞）
  - 輸入框使用泰文字型顯示，讓用戶確認輸入正確
  - 無字數硬上限，但建議每次分析 50 詞以內以確保回應速度
- **v1 不支援**：相機 OCR、語音輸入（列為 v2 功能）

### 分詞邏輯
- 泰語書寫系統沒有詞間空格，分詞由 OPENAI API 在語義層面判斷
- OPENAI 同時處理分詞與分析，不需要額外的 NLP 服務（目前無後端）
- 分詞結果以視覺間距呈現，用戶可一眼看出 App 如何切割詞彙

### API 呼叫設計
- **模型**：OPENAI gpt-5（成本低、速度快，分析準確度對此用途足夠）
- **單次請求**：一個 API call 完成所有事——分詞 + 每詞的拼音、詞義、聲調、拼讀分解
- **輸出格式**：嚴格 JSON array，每個元素對應一個詞塊
- **System Prompt 關鍵約束**：
  - `tone_reason` 欄位只能描述「視覺上可見的符號或字母形狀」，禁止使用「高類/中類/低類輔音」等術語
  - `zh_approx` 必須用中文近似音舉例，而非英語音標

### 本地快取
- 用 `SHA-256(thai_input)` 作為 key，將 API 回傳的完整 JSON 儲存於本機 SQLite / CoreData
- 同樣的文字再次輸入時，直接讀取快取，不重複呼叫 API
- 快取無過期機制（泰語詞彙的發音不會改變）
- 預期效果：反覆練習的用戶實際 API 花費接近零

### 錯誤處理
- 網路失敗時：顯示「無法連線，請稍後再試」，保留輸入內容
- 非泰文輸入時：Claude API 會自動偵測並回傳空分詞，App 顯示提示「請輸入泰文」
- API key 無效時：導引用戶至設定頁面重新設定

## Feature 2：四層垂直對齊顯示

### 概述
解析完成後，每個詞塊以垂直排列的四層資訊呈現。這是整個 App 最基本的閱讀視圖，也是所有互動的起點。

### 四個資訊層（由上至下）
```
ข้าว          ← 第一層：泰文原字（聲調顏色染色）
khâo          ← 第二層：RTGS 羅馬拼音
飯            ← 第三層：中文詞義
──────        ← 第四層：點擊後展開拼讀卡片
```

### 設計細節
- **泰文原字**：使用泰文字型（Noto Serif Thai 或系統泰文字型），字號充分大（iOS 上建議 28–32pt），確保泰文的聲調符號、元音符號清晰可見
- **羅馬拼音**：使用等寬字型（DM Mono 或 SF Mono），視覺上與泰文明顯區隔
- **中文詞義**：顯示最常見的詞義；若詞義有多個，預設顯示最貼近當前語境的一個
- **詞塊間距**：每個詞塊之間有適當間距，讓句子可以自然換行，不強制單行顯示
- **整體排版**：詞塊水平排列，句子過長時自然換行（wrap），維持閱讀語流

### 語法輔助標記（可選顯示）
- 語氣詞（ครับ / ค่ะ / นะ / เลย 等）以特殊標記區分，並在詞義旁加上括號說明：「(語氣詞，類似「啊/呢/嘛」)」
- 第一版實作為固定標記，不依據語境動態判斷語氣強度

## Feature 3：聲調顏色系統

### 概述
這是最核心的視覺差異化功能。泰語五個聲調各對應一個固定顏色，全文染色，用戶掃視一眼就能感知整篇文字的聲調分佈，不需要任何懸停或點擊。

### 五聲調顏色對照

| 聲調 | 泰語名稱 | 聲調符號 | 顏色 | 選色邏輯 |
|------|---------|---------|------|---------|
| 中調 สามัญ | 無符號 | 灰色 | 平穩，無方向感，不引人注意 |
| 低調 เอก | ่ | 藍色 | 沉，壓，往下 |
| 降調 โท | ้ | 紅色 | 急降，警示，強烈 |
| 高調 ตรี | ๊ | 橘黃色 | 高亢，熱，刺激 |
| 升調 จัตวา | ๋ | 綠色 | 上升，生長，清新 |

**設計原則**：顏色本身要暗示方向。用戶不需要記憶「藍色 = 低調」，只要感覺「藍色往下沉」就夠了。隨著使用頻率增加，顏色-聲調的連結會自然固化。

### 圖例顯示規則
- 圖例常駐於畫面固定位置（建議：文字區域上方，工具列下方），永遠可見
- 圖例顯示五個色點 + 聲調名稱 + 方向箭頭（例：`● 急降 ↘↘`）
- 圖例可折疊（對熟悉用戶減少視覺干擾），但折疊狀態要有一個小色點提示列存在
- 圖例不可完全隱藏（它是新用戶的即時對照表）

### 技術實作
- 聲調資料由 Claude API 分析輸出（`tone` 欄位，值為 `mid / low / fall / high / rise`）
- iOS 渲染層根據 `tone` 值套用對應的 `UIColor` 或 SwiftUI `Color`
- 顏色套用在**泰文原字**上；羅馬拼音和中文詞義保持預設顏色（灰黑），避免視覺過載

## Feature 4：點擊詞塊 → 拼讀卡片

### 概述
點擊任何一個詞塊，彈出一張拼讀分析卡片，逐層說明這個詞的構成——聲母、韻母、韻尾各是什麼音、怎麼發、像中文什麼聲音——以及聲調為何是這個調。

### 卡片結構（固定四區塊）

**區塊一：詞頭資訊**
- 泰文原字（大字，聲調顏色）
- RTGS 羅馬拼音
- 中文詞義

**區塊二：拼讀分解（核心）**
每個組成部分一行，格式如下：
```
聲母  ข  →  /kh/  送氣音，像「可」的 kh-
韻母  า  →  /aː/  長元音，像「啊——」
韻尾  ว  →  /-o/  收尾滑音（非靜音）
```
- `聲母`：泰文字母 + 對應 IPA 音標 + 中文近似音描述
- `韻母`：元音符號（可能是上方/下方/左右的符號組合）+ 長短標記 + 中文近似音
- `韻尾`：若有韻尾，標注其音 + 是否為「靜音韻尾」（閉音節）

**靜音韻尾的特殊處理**：
- 泰語閉音節（以 -บ/-ด/-ก 等結尾）的韻尾在口語中幾乎不發出聲音
- 但這個韻尾決定了音節的長短與聲調，必須顯示
- 標注方式：「停在嘴型上，不送氣」——對廣東話母語者尤其直觀（廣東話入聲字同樣機制）
- 用淡色 + 斜體文字標注，視覺上與「有發音的韻尾」區分

**區塊三：聲調說明**
- 顏色標籤（與全文染色系統一致）
- 聲調名稱 + 方向箭頭
- `tone_reason`：一行簡短說明，**只描述視覺上可見的東西**，不使用術語

**設計原則範例**：
- ✓ 正確：「้ 這個符號寫在這裡 → 音往下降」
- ✓ 正確：「ก 這個字母 + 開音節 → 平穩中調」
- ✗ 錯誤：「中類輔音在開音節無聲調符號時發中調」

**區塊四：播放按鈕**
- 呼叫 iOS AVSpeechSynthesizer，語言設定 `th-TH`
- 播放整個詞（而非逐字母播放）
- 按鈕旁標注「TTS 合成音，聲調僅供參考」——這是對用戶誠實的設計

### 互動細節
- 再次點擊同一個詞 → 卡片收合
- 點擊其他詞 → 卡片切換至新的詞
- 卡片出現時帶有短暫的滑入動畫（100–150ms），不影響閱讀流暢性
- 卡片不以 Modal 方式全屏覆蓋，而是內嵌在句子下方，用戶可以同時看到句子和卡片

## Feature 5：學習模式開關

### 概述
一個位於畫面右上角的開關，切換整個介面的資訊密度。解決「老用戶想要乾淨視圖 vs 新用戶需要完整拆解」的需求衝突，而不是為兩種用戶設計兩個 App。

### 關閉狀態（預設）

**適合**：快速查詢、旅行中使用、已有基礎的用戶。

每個詞塊只顯示三行：
```
ข้าว    ← 泰文（聲調染色）
khâo    ← 羅馬拼音
飯      ← 中文詞義
```

點擊任一詞 → 彈出拼讀卡片（詳見 Feature 4）。用戶主動求知，被動時介面保持乾淨。

### 開啟狀態（學習模式）

**適合**：刻意練習、初學者、想慢讀理解每個字的用戶。

介面切換為**卡片堆疊**排版（非橫向排列），每個詞一張卡片，卡片完整展開：

```
┌─────────────────────────────────┐
│  ข้าว  khâo  飯                  │
│  ─────────────────────────────  │
│  聲母  ข  →  /kh/  送氣 kh-      │
│  韻母  า  →  /aː/  長元音        │
│  韻尾  ว  →  /-o/  收尾滑音      │
│  ─────────────────────────────  │
│  🔴 降調  ้ 這個符號 → 音往下降   │
└─────────────────────────────────┘
```

**排版切換原因**：
- 橫向排列在開啟學習模式後每個詞塊的高度會急劇增加，橫向滑動閱讀非常不舒適
- 改為縱向卡片堆疊，用戶上下捲動，每張卡片資訊完整、不擁擠

### 開關記憶
- 用戶設定的開關狀態儲存於 UserDefaults，App 重啟後維持上次設定
- 預設為關閉（乾淨視圖）

## Feature 6：發音播放（TTS）

### 概述
讓用戶聽到泰文的實際發音，補足視覺顏色系統無法傳達的聲調細節。v1 使用 iOS 系統內建 TTS，不依賴外部服務。

### 播放層級
- **詞塊播放**：在拼讀卡片底部點擊「播放」，播放該單詞
- **全句播放**（v1 選做）：在句子視圖右上角加一個「▶ 全句」按鈕，播放整段文字

### TTS 設定
- 語言：`th-TH`
- 語速：預設稍慢（`AVSpeechUtteranceDefaultSpeechRate * 0.85`），讓用戶清楚聽出聲調輪廓
- 未來可加入語速滑桿讓用戶自調

### TTS 的誠實設計
iOS TTS 的泰語語音在聲調上存在不準確之處——這是技術現實，不掩蓋。所有播放按鈕旁都標注：「TTS 合成音，聲調僅供參考」。視覺聲調顏色系統是 v1 的聲調學習主力，TTS 是輔助。

### v2 升級路徑
待有資源後，可替換為：
- 授權第三方真人錄音資料庫（如 Forvo API）
- 自錄高品質泰語母語者錄音（針對最高頻詞彙優先）
- Thai Whisper

## Feature 7：API Key 管理（個人版）

### 概述
v1 為個人使用設計，不需要後端。用戶在 App 設定中輸入自己的 OPENAI API Key，所有請求直接從 iOS App 發出。

### 設定頁面
- 路徑：設定 → API 設定
- 輸入欄位：API Key（以密碼欄形式顯示，預設遮蔽）
- 驗證按鈕：輸入後可點擊「驗證」，發送一個最小測試請求確認 Key 有效
- 儲存位置：iOS Keychain（安全儲存，不存入 UserDefaults）

### 安全考量
- Key 只存在本機 Keychain，不上傳任何伺服器
- App 不記錄任何用戶輸入的泰文內容
- 未來公開版本需在中間加一個 Key Proxy 後端，避免 Key 暴露於用戶端

## 技術架構總覽

```
iOS App (SwiftUI)
│
├── 輸入層
│   └── TextEditor + 貼上偵測
│
├── 分析層
│   ├── 快取查詢 (CoreData, key = SHA-256)
│   └── Claude API (claude-haiku, 單次請求, JSON output)
│
├── 渲染層
│   ├── 句子視圖 (HStack + wrap layout)
│   ├── 聲調染色 (Color map from tone enum)
│   ├── 拼讀卡片 (Sheet or inline expand)
│   └── 學習模式 (VStack card stack)
│
└── 音頻層
    └── AVSpeechSynthesizer (th-TH)
```

## v1 功能邊界（明確不做）

以下功能刻意排除在 v1 之外，原因附上：

| 功能 | 排除原因 |
|------|---------|
| OCR 相機輸入 | 增加 Vision framework 依賴，v1 先驗證核心閱讀體驗 |
| 手動調整分詞 | Claude 分詞準確率對 v1 個人使用已足夠 |
| 語序對比標注 | 對入門用戶有幫助，但增加 UI 複雜度，v2 再加 |
| 帳號系統 / 同步 | 純個人工具不需要 |
| 課程 / 進度追蹤 | 刻意不做——這是工具，不是 Duolingo |
| 離線模式 | 依賴 API，暫無法離線 |
| Android | 先驗證 iOS |

## v2 潛在功能清單（供未來參考）

- OCR 相機輸入（菜單、招牌、路標）
- 語序對比視覺化（中泰語序箭頭標注）
- 生字本（長按任意詞加入收藏，形成個人詞庫）
- 語境詞義展開（多義詞根據句子語境說明現在是哪個用法）
- TTS 替換為真人錄音
- 語速控制滑桿
- 分享功能（將分析結果截圖或輸出為圖片）
- 後端 Key Proxy（為公開版本準備）
